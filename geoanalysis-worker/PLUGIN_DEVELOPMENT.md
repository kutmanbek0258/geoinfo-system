# Руководство по разработке плагинов для GeoAnalysis Worker

Данный документ описывает стандарты, архитектурные ограничения и процесс создания новых аналитических модулей (плагинов) для сервиса `geoanalysis-worker`.

## 🧩 Архитектура "Ядро и Плагины"

Воркер разделен на две части:
1.  **Core (Ядро):** Инфраструктурный слой. Отвечает за Kafka, MinIO, создание временных папок в RAM-диске (`tmpfs`) и очистку ресурсов.
2.  **Plugins (Плагины):** Изолированные модули с бизнес-логикой. Плагин "не знает" о существовании сети, баз данных или облачного хранилища. Он работает только с локальными файлами в предоставленной ему директории.

---

## 🛠 Процесс создания плагина

### 1. Создание файла
Все плагины должны располагаться в директории `app/plugins/`. Имя файла должно быть осмысленным (например, `buffer_analysis.py`).

### 2. Наследование и реализация
Ваш класс должен наследоваться от `GeoWorkerPlugin` и реализовывать два обязательных элемента:
*   Свойство `plugin_name` (строка, по которой ядро находит плагин).
*   Метод `run()` (основная логика).

### 3. Шаблон кода
```python
from typing import Dict, Any
from ..core.base_plugin import GeoWorkerPlugin
from ..core.config import logger

class MyNewAnalysisPlugin(GeoWorkerPlugin):
    @property
    def plugin_name(self) -> str:
        return "my_analysis_name"  # Это имя будет использоваться в Kafka

    def run(self, local_inputs: Dict[str, str], params: Dict[str, Any], workspace: str) -> Dict[str, str]:
        # 1. Извлечение входных путей
        input_path = local_inputs.get("input_file_key")
        
        # 2. Извлечение параметров
        buffer_size = params.get("buffer_size", 100)
        
        # 3. Путь для результата (обязательно внутри workspace!)
        output_path = os.path.join(workspace, "result.geojson")
        
        # 4. Ваша логика (GDAL, Geopandas, и т.д.)
        # ... выполнение анализа ...
        
        # 5. Возврат словаря с результатами (ключи будут именами файлов в S3)
        return {"analysis_result": output_path}
```

## 🚫 Жесткие ограничения (Safety Rules)

Чтобы воркер оставался стабильным и безопасным, при разработке плагинов **ЗАПРЕЩЕНО**:

1.  **Сетевые вызовы:** Никаких `requests.get()`, обращений к БД или внешним API. Если плагину нужны данные — они должны быть переданы как входные файлы через MinIO.
2.  **Работа вне Workspace:** Плагин может читать и писать файлы **только** внутри пути, переданного в аргументе `workspace`.
3.  **Глобальное состояние:** Плагины должны быть stateless. Не используйте глобальные переменные для хранения данных между запусками.
4.  **Утечки GDAL:** При использовании GDAL/OGR всегда закрывайте датасеты (`ds = None`), чтобы избежать блокировки файлов и переполнения памяти.

---

## 💾 Работа с памятью (tmpfs)

Воркер использует RAM-диск. Это очень быстро, но объем ограничен (обычно 4ГБ).
*   Старайтесь использовать потоковую обработку или генераторы для больших данных.
*   Если ваш алгоритм требует объема RAM больше лимита, он будет принудительно завершен системой (OOM).
*   Всегда указывайте GDAL использовать `workspace` для временных файлов:
    `gdal.SetConfigOption('CPL_TMPDIR', workspace)`

## 🧪 Тестирование

Перед деплоем проверьте плагин локально, передав ему имитацию локальных путей:

```python
plugin = MyNewAnalysisPlugin()
results = plugin.run(
    local_inputs={"input_file_key": "test_data/input.tif"},
    params={"buffer_size": 50},
    workspace="./test_output"
)
print(results)
```

## 📋 Системные и устанавливаемые плагины (Schema-Driven UI)

В ГИС-системе плагины делятся на два архитектурных типа:

1.  **Системные плагины (System Plugins):**
    *   Не имеют динамической UI-схемы (метод `get_schema()` возвращает `None` или отсутствует).
    *   Для них во фронтенде написаны жестко закодированные индивидуальные Vuetify-диалоги.
2.  **Устанавливаемые плагины (Installed/Dynamic Plugins):**
    *   Должны реализовывать метод `get_schema()`, возвращающий валидную JSON-схему (**JSON Schema Draft-07**) для генерации UI.
    *   Фронтенд автоматически рендерит форму запуска через универсальный компонент `DynamicAnalysisDialog.vue`.

### Пример реализации метода `get_schema()`:
```python
    def get_schema(self) -> Optional[dict]:
        import json
        import os
        schema_path = os.path.join(os.path.dirname(__file__), "my_plugin.schema.json")
        with open(schema_path, "r", encoding="utf-8") as f:
            return json.load(f)
```

Каждая динамическая схема должна использовать ГИС-форматы для пространственных полей ввода:
*   `"format": "raster-layer"` — выбор растрового слоя.
*   `"format": "vector-layer"` — выбор векторного слоя.
*   `"format": "terrain-layer"` — выбор цифровой модели рельефа (DEM).
*   `"format": "map-point"` — выбор координат точки кликом на карте.
*   `"format": "project-document"` — выбор прикрепленного документа проекта (например, чертежей DXF).
*   `"format": "vector-field-select"` — выбор поля атрибутов векторного слоя.

## 📡 Протокол динамической регистрации схем через Kafka

Обмен схемами между `geoanalysis-worker` и `geoabstraction-service` происходит по следующему протоколу:

1.  **Запрос бэкенда (`geoanalysis.schema.request`):**
    При запуске или сбросе кэша Java-бэкенд отправляет событие:
    ```json
    { "action": "GET_SCHEMAS", "timestamp": "..." }
    ```
2.  **Обработка воркером:**
    Фоновый слушатель оркестратора принимает событие и сканирует все загруженные классы плагинов.
3.  **Ответ воркера (`geoanalysis.schema.response`):**
    Для каждого плагина, вернувшего непустую схему, воркер отправляет отдельное сообщение:
    ```json
    {
      "pluginName": "my_plugin",
      "title": "Мой плагин",
      "icon": "mdi-puzzle-outline",
      "schema": { ...содержимое JSON-схемы... },
      "timestamp": "..."
    }
    ```
    Бэкенд сохраняет эти схемы в СУБД для раздачи по REST API во фронтенд и для валидации параметров задач.
