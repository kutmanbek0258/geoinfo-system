# GeoAnalysis Worker (В процессе разработки)

## 📌 Обзор

**GeoAnalysis Worker** — это высокопроизводительный аналитический сервис на базе Python, предназначенный для выполнения тяжелых пространственных операций над геоданными. Он является частью экосистемы **GeoInfoSystem** и отвечает за такие задачи, как генерация изолиний (contours), создание буферных зон, пространственные пересечения и анализ растров.

> **Статус:** 🚧 Сервис находится в активной фазе разработки и проектирования.

## 🏗 Архитектурные принципы

Воркер спроектирован как изолированная «числодробилка» (number cruncher), следуя двум ключевым паттернам:

1.  **Core & Plugins (Ядро и Плагины):** Основная логика (оркестрация, связь с Kafka и S3) отделена от аналитических алгоритмов. Новые типы анализа добавляются простым созданием нового плагина.
2.  **Stateless RAM Processing (tmpfs):** Для максимальной производительности и минимизации износа дисков все промежуточные операции чтения-записи выполняются в оперативной памяти через монтирование `tmpfs`.

## ⚙️ Принцип работы (Пайплайн)

Взаимодействие с воркером происходит полностью асинхронно через брокер сообщений **Kafka** и объектное хранилище **MinIO**.

1.  **Listen (Ожидание):** Оркестратор слушает топик `geoabstraction.tasks`. При получении сообщения (задачи) извлекается `taskId` и `pluginName`.
2.  **Setup (Подготовка):** В RAM-диске (`/app/workspace`) создается уникальная временная директория `task_{UUID}`.
3.  **Pull (Загрузка):** Ядро скачивает необходимые файлы (растры, DEM, векторы) из MinIO напрямую в подготовленную папку в RAM.
4.  **Execute (Выполнение):** Управление передается соответствующему плагину. Плагин выполняет вычисления, используя GDAL, OGR или специализированные Python-библиотеки, сохраняя результат в ту же временную папку.
5.  **Push (Выгрузка):** Ядро берет созданные артефакты и загружает их обратно в MinIO в корзину обработанных данных.
6.  **Report (Отчет):** В топик `geoabstraction.results` отправляется JSON-ответ со статусом `COMPLETED`, ссылками на новые файлы в S3 и метриками выполнения (время, потребление памяти).
7.  **Cleanup (Очистка):** Временная директория в `tmpfs` безвозвратно удаляется, освобождая оперативную память.

## 🛠 Технологический стек

*   **Язык:** Python 3.10+
*   **ГИС-ядро:** GDAL/OGR 3.6+
*   **Библиотеки:** 
    *   `kafka-python` — взаимодействие с брокером.
    *   `minio` — работа с S3-хранилищем.
    *   `rasterio`, `geopandas`, `shapely` — (планируется) для сложной пространственной логики.
*   **Инфраструктура:** Docker с лимитами ресурсов и RAM-диском.

## 🚀 Развертывание

Воркер запускается в составе общего Docker Compose проекта. Важной особенностью является конфигурация `tmpfs`:

```yaml
geoanalysis-worker:
  image: geoinfosystem/geoanalysis-worker
  tmpfs:
    - /app/workspace:size=4G,mode=1777 # RAM-диск для операций
  deploy:
    resources:
      limits:
        memory: 6G # 4G для tmpfs + 2G для процессов
  environment:
    - KAFKA_BOOTSTRAP_SERVERS=kafka:9092
    - MINIO_ENDPOINT=minio:9000
    - MINIO_ACCESS_KEY=${MINIO_ROOT_USER}
    - MINIO_SECRET_KEY=${MINIO_ROOT_PASSWORD}
```

## 🧩 Разработка новых плагинов

Каждый плагин должен наследоваться от базового класса и реализовывать метод `run()`. Плагинам **запрещено**:
*   Напрямую обращаться к внешним API или базам данных.
*   Использовать пути вне предоставленного `workspace`.

Все данные для работы плагин должен получать в виде локальных путей к файлам, которые ядро предварительно скачало из S3.

## ⚠️ Безопасность и отказоустойчивость

*   **Timeout:** Каждая задача выполняется с ограничением по времени. Если плагин "завис", процесс принудительно убивается, а RAM очищается.
*   **Sandboxing:** Воркер не имеет доступа к PostGIS или API Gateway, что исключает возможность компрометации основных данных системы через аналитические скрипты.
