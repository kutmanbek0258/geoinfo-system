# **Архитектурное решение: Оптимизация редактирования тяжелых векторов (MVT/GeoJSON) V2**

Настоящий документ определяет обновленную техническую стратегию и UX/UI-сценарии для оптимизации работы с пространственными данными в веб\-клиенте ГеоИнфоСистемы при редактировании тяжелых векторных объектов (MultiPolygon).

## **1\. Анализ проблемы (MVT vs GeoJSON Bottleneck)**

В текущей реализации ГеоИнфоСистемы рендеринг миллионов объектов в режиме просмотра эффективно решен с помощью векторных тайлов Mapbox Vector Tiles (MVT) через pg\_tileserv. Однако при переходе в режим редактирования загрузка тяжелых мультиполигонов (тысячи контуров/островков, сотни тысяч вершин) целиком в формате GeoJSON вызывает критическое падение производительности веб\-клиента (зависание UI-потока JavaScript).  
Основными причинами зависания являются:

* **Парсинг JSON:** Десериализация тяжелой текстовой строки GeoJSON полностью блокирует основной поток (Main Thread) браузера.  
* **Оверхед OpenLayers:** Создание тысяч объектов внутренних геометрий, вычисление их ограничивающих рамок (BBox) и инициализация интерактивных «ручек» (vertex handles) для редактирования перегружают оперативную память клиента.

## **2\. Сравнение альтернативных вариантов решения**

| Подход | Механизм реализации | Плюсы | Минусы |
| :---- | :---- | :---- | :---- |
| **А. Декомпозиция по требованию (ST\_Dump)** | Разделение MultiPolygon на составляющие Polygon на сервере. Клиент редактирует только выбранный элемент. | Минимальный объем данных в браузере, изоляция изменений, полное отсутствие лагов. | Требует дополнительной логики сборки на бэкенде при сохранении. |
| **Б. Пространственный клиппинг (ST\_Intersection)** | Обрезка геометрии на сервере по текущему Bounding Box (экрану) пользователя перед отправкой GeoJSON. | Браузер получает только то, что видит на экране. Высокая скорость загрузки. | Искажает топологию объекта за пределами экрана; сложно собирать обратно без швов. |
| **В. Переход на FlatGeobuf** | Замена GeoJSON на бинарный протокол FlatGeobuf с поддержкой стриминга и пространственных индексов. | Парсинг в разы быстрее, возможность читать только нужные байты из файла. | Не решает проблему перегрузки OpenLayers при отрисовке вершин для модификации. |

## **3\. Целевая стратегия: Гибридный паттерн «Shot Frame» \+ Декомпозиция**

Оптимальным решением является объединение пространственной фильтрации на стороне клиента и структурной декомпозиции геометрии на стороне сервера. Вместо загрузки всего мультиполигона, система переходит в режим ожидания фокуса пользователя, фиксирует область видимости («кадр снимка» / Shot Frame) с временной задержкой (debounce) и запрашивает у geodata-service только те составные части (Polygon) объекта, которые пересекают данный экранный экстент.

## **4\. Пошаговый UX/UI Сценарий работы пользователя**

1. **Инициализация и подсказка (Toast Message):** Пользователь выбирает векторный объект на карте (отображаемый через MVT) и нажимает кнопку «Изменить геометрию». На экране появляется всплывающее уведомление (Toast): *«Пожалуйста, приблизьтесь к редактируемой области и подождите 1 секунду для загрузки точек»*. Кнопка подтверждения изменений блокируется (disabled).  
2. **Контроль масштаба и задержка (Debounce):** Пользователь приближает карту (Zoom In) к целевому участку тяжелого мультиполигона. Система активирует логику только при достижении рабочего масштаба (zoom \>= 16). Запускается таймер задержки в 1000 мс. При сдвиге или зуме карты таймер сбрасывается.  
3. **Фиксация кадра («Shot Frame»):** При остановке карты на 1 секунду, фронтенд визуально подсвечивает прямоугольную рамку (границы текущего экрана) и отправляет асинхронный запрос на бэкенд, передавая идентификатор объекта и координаты Bounding Box.  
4. **Серверная выборка и нарезка:** Микросервис geodata-service разворачивает исходный геообъект через ST\_Dump, фильтрует части с помощью ST\_Intersects по переданному BBox и возвращает компактный GeoJSON, где в свойствах (properties) каждого полигона указан его уникальный внутренний индекс (sub\_id).  
5. **Переход в режим интерактивных вершин (Red Dots):** На фронтенде внутри Shot Frame рендерятся ребра полученных полигонов, а их вершины подсвечиваются яркими красными точками (интерактивными узлами редактирования OpenLayers). Пользователь видит, что объект готов к модификации. Подсказка меняется на: *«Перетаскивайте красные точки для изменения. Для сохранения нажмите "Подтвердить"»*.  
6. **Интерактивное редактирование:** Пользователь перетаскивает красные точки, добавляет новые узлы или удаляет существующие. OpenLayers отслеживает, какие именно составные части (по sub\_id) были затронуты изменениями.  
7. **Фиксация и отправка изменений:** Пользователь нажимает кнопку «Подтвердить изменения». На бэкенд отправляется точечный пакет обновлений, содержащий массив только измененных частей геометрии с их sub\_id. Появляется спиннер загрузки. После успешного ответа сервера режим редактирования закрывается, Shot Frame исчезает, а кэш MVT тайлов в Nginx инвалидируется.

## **5\. Техническая спецификация реализации (PostGIS / Java / OpenLayers)**

### **5.1. Запрос списка составных частей в рамках BBox**

SELECT   
    (parts.path)\[1\] as sub\_feature\_id,  
    ST\_AsGeoJSON(parts.geom) as geojson\_part  
FROM (  
    SELECT (ST\_Dump(geometry)).\*   
    FROM main\_geotable   
    WHERE id \= :id  
) as parts  
WHERE ST\_Intersects(parts.geom, ST\_MakeEnvelope(:minX, :minY, :maxX, :maxY, 4326));

### **5.2. Атомарное обновление модифицированной части на бэкенде (Spring Boot)**

\-- Обновление N-й геометрии внутри мультиполигона  
UPDATE main\_geotable  
SET geometry \= ST\_SetGeometryN(geometry, :sub\_feature\_id, ST\_GeomFromText(:new\_polygon\_wkt, 4326))  
WHERE id \= :id;

## **6\. Влияние на смежные компоненты системы**

* **QGIS Connector (Плагин):** Не затрагивается изменениями, так как настольный клиент использует полноценный GDAL/OGR стек и пакетную синхронизацию, легко справляясь с тяжелыми MultiPolygon на уровне ядра C++. Изменения изолированы внутри веб\-интерфейса Vue.js.  
* **Nginx Proxy (Кэширование):** Инвалидация кэша MVT тайлов в Nginx после сохранения изменений по-прежнему выполняется по ID родительского объекта, гарантируя обновление карты у всех пользователей.
