# Стратегия изоляции динамических плагинов геоаналитики на базе виртуальных окружений (venv)

## 1. Введение и концептуальный выбор

Переход платформы ГеоИнфоСистемы к модели динамической загрузки аналитических модулей через Web UI требует кардинального пересмотра управления Python-зависимостями. Поскольку плагины могут разрабатываться изолированно и использовать специфические сторонние библиотеки (например, кастомные пакеты ML, специализированные математические утилиты), их установка непосредственно в базовое системное окружение контейнера `geoanalysis-worker` создает критические риски:

* **Конфликты версий:** Обновление или замена транзитивных зависимостей может нарушить работу базовых тяжеловесных ГИС-библиотек (GDAL, GeoPandas, Rasterio, Shapely).
* **Нарушение стабильности (OOM / Сраши):** Падение или утечка памяти в одном динамическом модуле могут привести к аварийной остановке всего воркера и обрыву конвейера Kafka.

Для минимизации рисков выбрана **стратегия изолированных виртуальных окружений (venv) в сочетании с субпроцессной моделью выполнения**. Это позволяет рантайму оставаться полностью динамическим (Hot-Reloading), предсказуемым и безопасным.

## 2. Модель выполнения: Изолированный субпроцесс (Subprocess Mode)

> **Архитектурный сдвиг:** Вместо динамического импорта кода плагина напрямую в адресное пространство основного процесса оркестратора `geoanalysis-worker` через `importlib`, оркестратор выполняет задачу в **отдельном операционном процессе**, вызывая бинарный файл Python конкретного `venv` плагина.

### Преимущества субпроцессного venv-выполнения:

1. **Абсолютная изоляция памяти:** Если алгоритм (например, расчет водосборов или зональная статистика) превысит лимиты RAM, Linux OOM-Killer уничтожит только этот изолированный дочерний процесс. Основной контейнер воркера и слушатель Kafka продолжат работу.
2. **Отсутствие утечек в sys.modules:** При удалении или обновлении плагина нет необходимости очищать кэш импорта Python в основном процессе, что исключает фрагментацию памяти и "зависание" старого кода.
3. **Естественная активация окружения:** Запуск скрипта через `/app/plugins/dynamic/<plugin_id>/.venv/bin/python` автоматически изолирует переменные среды, пути поиска пакетов (`sys.path`) и контекст выполнения.

## 3. Жизненный цикл плагина в системе (Lifecycle Pipeline)

Управление плагинами автоматизировано и разделено между Java-бэкендом и Python-воркером.

| Этап | Действия на стороне бэкенда / воркера | Результат |
| --- | --- | --- |
| **1. Загрузка и Валидация** | Пользователь загружает ZIP в Web UI. Бэкенд `geoabstraction-service` распаковывает его в памяти, проверяет наличие `main.py`, `schema.json` и синтаксис JSON-схемы Draft-07. Ссылки сохраняются в бакет MinIO `gis-plugins`. | Файл зафиксирован в S3; метаданные готовы к публикации. |
| **2. Оркестрация (Kafka)** | Бэкенд отправляет событие `INSTALL` в топик `geoanalysis.plugin.lifecycle`. Все реплики воркеров вычитывают событие. | Синхронизация состояния кластера воркеров. |
| **3. Развертывание venv** | Каждый воркер скачивает ZIP, распаковывает в `/app/plugins/dynamic/<plugin_id>/`. Запускается команда создания venv с наследованием системных пакетов. С помощью `pip` доустанавливаются зависимости из `requirements.txt`. | Сформировано изолированное рабочее окружение плагина. |
| **4. Регистрация UI** | Воркер считывает локальный `schema.json` и отправляет его структуру обратно в топик `geoanalysis.schema.response`. Бэкенд обновляет таблицу `analysis.plugin_schemas`. | Интерфейс динамически обновляется на карте у пользователей. |
| **5. Выполнение** | При получении задачи на расчет, оркестратор воркера вызывает python-интерпретатор из venv плагина в режиме субпроцесса. | Безопасный расчет с изоляцией ресурсов. |
| **6. Удаление (Uninstall)** | Событие `UNINSTALL` через Kafka очищает директорию плагина с диска воркера. Бэкенд удаляет схему из БД. | Полная очистка диска и UI без даунтайма. |

## 4. Оптимизация сборки: Наследование системных пакетов

Ключевая проблема чистых `venv` — это необходимость компиляции тяжелых бинарных C-extensions (например, GDAL, Rasterio, NumPy, SciPy) при каждой установке плагина, что может занимать до 10-15 минут и требовать компиляторов в рантайме. Это неприемлемо для концепции быстрого внедрения плагина.

> **Решение: Флаг `--system-site-packages**`
> Виртуальное окружение каждого динамического плагина создается с явным указанием наследования глобальных пакетов:
> ```bash
> python3 -m venv --system-site-packages /app/plugins/dynamic/<plugin_id>/.venv
> 
> ```
>
>

Базовый Docker-образ `geoanalysis-worker` уже содержит в себе протестированные, скомпилированные и стабильные версии тяжелого картографического стека. При создании venv с этим флагом плагин **мгновенно** получает доступ к системным GDAL и GeoPandas, а через свой локальный `requirements.txt` доустанавливает исключительно легковесные библиотеки (например, `scikit-learn`, `requests`, `scipy` новых версий), что занимает не более 2-5 секунд.

## 5. Безопасность и Валидация (Песочница)

Загрузка исполняемого кода через Web UI несет критический риск удаленного вызова кода (RCE) и компрометации инфраструктуры. Стратегия включает пять оборонительных рубежей:

### 5.1. Статический анализ структуры и зависимостей пакета

Внутри бэкенда перед отправкой плагина в MinIO проверяются лимиты: размер ZIP не более 50 МБ, объем распакованных данных не более 200 МБ. Файлы конфигураций ОС, исполняемые бинарные файлы (`.sh`, `.exe`, `.so`) блокируются на этапе разархивации.
* **Валидация зависимостей:** Бэкенд выполняет статический разбор `requirements.txt` и блокирует установку, если плагин пытается обновить или переопределить ключевые системные ГИС-библиотеки (список: `gdal`, `rasterio`, `shapely`, `numpy`, `geopandas`), во избежание коллизий и падения базового картографического слоя.

### 5.2. Ограничение прав процесса выполнения и Сетевая изоляция

Субпроцесс плагина вызывается от лица несистемного пользователя Linux (`gis_worker` с фиксированным UID `10001`) с жестко ограниченными правами доступа. 
* Дочернему процессу закрывается доступ на чтение системных конфигурационных файлов бэкенда (`/app/config`, `/app/.env`).
* С помощью флагов операционной системы или утилит контейнеризации дочернему процессу полностью запрещается доступ к сетевому интерфейсу, что исключает кражу учетных записей или несанкционированные внешние вызовы.

### 5.3. Защита RAM-диска (tmpfs) и Лимитирование ресурсов CPU/RAM

Каждому субпроцессу при старте передается изолированная поддиректория внутри RAM-диска `tmpfs` в качестве параметра `workspace`. 
* На уровне вызова субпроцесса задается лимит дискового пространства и квота на использование оперативной памяти (через системные вызовы `ulimit -v` или cgroups) во избежание DoS-атак на память хоста.
* **Конкурентное выполнение:** Оркестратор воркера ограничивает максимальное число параллельно выполняемых задач с помощью `ThreadPoolExecutor` с числом рабочих потоков, равным `max_workers = CPU_CORES`, предотвращая перегрузку процессора.

### 5.4. Унификация IPC через SDK-обертку (`gis-plugin-sdk`)

Для упрощения разработки и снижения риска ошибок ручного парсинга, логика взаимодействия (IPC) выносится в базовую Python-библиотеку `gis-plugin-sdk`:
* SDK автоматически парсит аргумент командной строки `--context`, считывает `task_context.json`, предоставляет плагину строго типизированный доступ к `inputs` и `parameters`, а также берет на себя безопасную запись результатов в `output_metadata.json` при выходе.

### 5.5. Версионирование и Логирование субпроцессов

* **Версионирование:** Событие `INSTALL` в топике Kafka содержит не только идентификатор плагина, но и его версию (например, `plugin_id: "viewshed_analysis_dynamic", version: "1.2.0"`). Это позволяет воркерам разворачивать плагины в изолированные папки версий `/app/plugins/dynamic/viewshed_analysis_dynamic/1.2.0/` и параллельно поддерживать разные версии или осуществлять бесшовный откат.
* **Перехват логов:** Потоки `stdout` и `stderr` запущенного субпроцесса перехватываются оркестратором и перенаправляются в системный логгер воркера с префиксом `[Plugin: <plugin_id>@<version>]`, обеспечивая полную прозрачность и удобную диагностику при падениях.

## 6. Практическая реализация: Шаблон запуска воркера

Ниже представлен концептуальный код оркестратора Python-воркера для развертывания и вызова плагина через venv-субпроцесс:

```python
import os
import subprocess
import json
from app.core.config import logger

def install_dynamic_plugin(plugin_id: str, zip_extracted_path: str):
    venv_path = os.path.join(zip_extracted_path, ".venv")
    logger.info(f"Создание venv с системными пакетами для {plugin_id}")
    
    # 1. Создание venv с доступом к глобальному GDAL/GeoPandas
    subprocess.run(["python3", "-m", "venv", "--system-site-packages", venv_path], check=True)
    
    # 2. Установка специфичных зависимостей плагина
    req_txt = os.path.join(zip_extracted_path, "requirements.txt")
    if os.path.exists(req_txt):
        pip_executable = os.path.join(venv_path, "bin", "pip")
        logger.info(f"Установка зависимостей из requirements.txt для {plugin_id}")
        subprocess.run([pip_executable, "install", "--no-cache-dir", "-r", req_txt], check=True)

def execute_plugin_subprocess(plugin_id: str, inputs: dict, params: dict, workspace: str) -> dict:
    plugin_base = f"/app/plugins/dynamic/{plugin_id}"
    venv_python = os.path.join(plugin_base, ".venv", "bin", "python")
    
    # Подготовка контекста задачи в виде временного JSON-файла внутри workspace
    task_context_path = os.path.join(workspace, "task_context.json")
    with open(task_context_path, "w", encoding="utf-8") as f:
        json.dump({"inputs": inputs, "parameters": params, "workspace": workspace}, f)
        
    # Вызов плагина как изолированного процесса. Файл main.py запускается модульно
    main_py = os.path.join(plugin_base, "main.py")
    
    logger.info(f"Старт субпроцесса плагина {plugin_id} в venv")
    result = subprocess.run(
        [venv_python, main_py, "--context", task_context_path],
        capture_output=True,
        text=True,
        timeout=600 # Жесткий тайм-аут на тяжелую операцию (10 минут)
    )
    
    if result.returncode != 0:
        logger.error(f"Ошибка выполнения плагина: {result.stderr}")
        raise RuntimeError(f"Plugin crash: {result.stderr}")
        
    # Считывание результатов, которые плагин записал в выходной файл в workspace
    output_meta_path = os.path.join(workspace, "output_metadata.json")
    with open(output_meta_path, "r", encoding="utf-8") as f:
        return json.load(f)

```

## 7. Пошаговый план внедрения стратегии

1. **Модификация Dockerfile воркера:** Убедиться, что виртуальное окружение самого Docker-контейнера позволяет использовать `python3 -m venv` и глобальный пользователь имеет права на запись в папку `/app/plugins/dynamic/`.
2. **Разработка жизненного цикла в Java-бэкенде:** Создание нового Kafka-топика контроля жизненного цикла, реализация контроллера загрузки ZIP, интеграция валидатора JSON-схем Draft-07 перед загрузкой в MinIO.
3. **Реализация оркестратора в Python:** Интеграция методов динамического развертывания (`subprocess.run` для pip) и субпроцессного вызова задач с перехватом `stdout/stderr`.
4. **Тестирование сценариев отказов:** Симуляция загрузки плагина с синтаксической ошибкой в коде, плагина с конфликтующей библиотекой в `requirements.txt`, и плагина, вызывающего искусственную утечку памяти (проверка OOM-изоляции).
