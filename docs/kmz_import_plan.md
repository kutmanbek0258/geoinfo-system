# План реализации импорта KMZ в geodata-service

## 1. Обзор задачи
KMZ — это сжатый (ZIP) KML-файл, который может содержать вспомогательные ресурсы, такие как иконки, изображения и стили. Текущая реализация поддерживает только чистые KML-файлы и внешние (HTTP) ссылки на иконки.

## 2. Архитектурные изменения

### 2.1. Рефакторинг KmlImportServiceImpl
Необходимо отделить логику парсинга KML от источника данных (`MultipartFile`), чтобы один и тот же код мог обрабатывать как обычный KML, так и KML, извлеченный из KMZ-архива.

**Шаги:**
- Создать метод `processKml(InputStream kmlStream, Project project, Map<String, byte[]> resources)`, где `resources` — это карта локальных файлов из архива (для KMZ).
- Унифицировать метод `uploadIcon`, чтобы он сначала искал иконку в локальных ресурсах (для KMZ), а затем пробовал скачать по HTTP.

### 2.2. Обработка ZIP-архива (KMZ)
- Использовать `java.util.zip.ZipInputStream` для распаковки.
- Основной файл в KMZ обычно называется `doc.kml`, но согласно спецификации это может быть любой `.kml` файл в корне или первый найденный.
- Все остальные файлы в архиве должны быть считаны в память (или временное хранилище) как `Map<String, byte[]>` для последующей привязки к объектам.

## 3. Этапы реализации

### Этап 1: Подготовительный
1.  Изучить структуру типовых KMZ файлов (особенно пути к иконкам: `files/icon.png`, `../images/etc`).
2.  Подготовить тестовые наборы данных (.kml и .kmz).

### Этап 2: Расширение логики импорта
1.  **Создание `KmzUnzipper`:**
    - Реализовать извлечение `doc.kml`.
    - Сбор всех ресурсов в `Map<String, byte[]>`.
2.  **Обновление `KmlImportServiceImpl`:**
    - Изменить сигнатуру методов парсинга для поддержки локальных ресурсов.
    - Добавить логику сопоставления путей в KML с файлами из KMZ.
    - Интеграция с `DocumentServiceClient` для загрузки извлеченных иконок.

### Этап 3: Обновление API
1.  **KmlImportController:**
    - Разрешить загрузку файлов с расширением `.kmz`.
    - Добавить проверку MIME-типа `application/vnd.google-earth.kmz`.
2.  **KmlImportService:**
    - Добавить метод `importKmz(...)` или сделать `importKml` универсальным, определяющим тип по расширению/содержимому.

### Этап 4: Тестирование
1.  **Unit-тесты:**
    - Тестирование распаковки KMZ.
    - Тестирование парсинга KML с относительными путями к ресурсам.
2.  **Интеграционные тесты:**
    - Полный цикл импорта KMZ через REST API.
    - Проверка сохранения геометрии в БД и иконок в MinIO.

## 4. Особенности реализации
- **Память:** KMZ файлы могут быть большими. Нужно ограничить размер обрабатываемого архива или использовать потоковую обработку, чтобы не переполнить Heap.
- **Безопасность:** Защита от Zip Slip уязвимости (проверка путей извлекаемых файлов).
- **Кодировки:** Обработка различных кодировок в KML (обычно UTF-8).

## 5. Ожидаемый результат
Пользователь может загрузить `.kmz` файл. Система автоматически распаковывает его, создает проект, импортирует все точки, линии и полигоны, сохраняя их стили и локальные иконки в объектном хранилище.
