# Архитектурно-техническое решение: Интеграция QGIS в экосистему GeoInfoSystem

## 1. Общие принципы и ограничения безопасности

Настольное приложение QGIS интегрируется в экосистему GeoInfoSystem в качестве тяжелого аналитического клиента. При этом строго соблюдаются следующие архитектурные правила:

* **Никакого прямого доступа к БД:** Взаимодействие с базой данных происходит исключительно через REST API соответствующих микросервисов.
* **Никакого прямого доступа к брокеру сообщений (Kafka):** Все уведомления в реальном времени и события синхронизации маршрутизируются через специализированный WebSocket-шлюз.
* **Централизованная безопасность (Zero Trust):** Проверка прав и идентификация осуществляются через механизмы JWT/OAuth2 на уровне API Gateway. Прямой доступ к объектному хранилищу (MinIO) без валидации прав запрещен.

## 3. Компоненты инфраструктуры и шина данных

### 3.1. Sync Service (WebSocket Gateway)

Специализированный сервис для обеспечения реактивного взаимодействия между сервером и настольными клиентами QGIS.

* **Функции:** Агрегация событий из Kafka (топики `geo.data.events`, `doc.data.events`, `geoabstraction.*`), их фильтрация с учетом прав доступа конкретного пользователя (запрос в `auth-service`) и доставка до плагина.
* **Протокол:** WebSocket (STOMP или чистый JSON).
* **Критические типы сообщений:**
* `LAYER_UPDATED`: Сигнал для QGIS на инвалидацию локального кэша и принудительную отрисовку изменившегося векторного/растрового слоя.
* `JOB_COMPLETED`: Уведомление о завершении асинхронной обработки тяжелых данных (например, расчет NDVI воркером завершен).
* `FOLDER_STRUCTURE_CHANGED`: Триггер для обновления дерева каталогов в интерфейсе плагина.



### 3.2. Подсистема доставки и стриминга растров

Для оптимизации сетевого трафика и обеспечения возможности ГИС-анализа применяется гибридный метод доставки растровых данных и матриц высот (DEM):

| Тип данных / Назначение | Используемый формат / Протокол | Механизм работы в QGIS |
| --- | --- | --- |
| **Базовая 2D-визуализация** | WMS / WMTS (через GeoServer & Nginx) | Обычная подложка карт. QGIS запрашивает готовые RGB-тайлы (PNG/JPEG) под текущий масштаб экрана. |
| **3D-визуализация рельефа** | Cesium Terrain (Quantized Mesh) | Нативный рендеринг трехмерной текстурированной поверхности в 3D-окне QGIS (версии 3.34+). Представляет собой оптимизированную сеть (TIN Mesh). |
| **Растровый и ГИС-анализ** | **COG (Cloud Optimized GeoTIFF)** | Исходная матрица высот/каналов. Считывается ядром GDAL через `/vsicurl/` с помощью частичных HTTP Range-запросов (скачиваются только необходимые пиксели для расчета). |

## 4. Интеграция с существующими API микросервисов

### 4.1. Навигация и Векторные данные (Geodata Service)

* **Дерево проектов:** Эндпоинт `/api/geodata/folders/project/{id}` отдает иерархическую структуру для построения панели слоев в QGIS.
* **Транзакционный CRUD:** Загрузка данных осуществляется через `GET /api/geodata/project/page-query` (с обязательной фильтрацией по BBox текущего охвата экрана QGIS). Сохранение изменений (геометрия и атрибуты) передается пакетами через `POST/PUT /api/geodata/project`.

### 4.2. Растровые абстракции (GeoData Service)

* **Каталог:** `GET /api/geodata/raster-layers/page-query` возвращает метаданные доступных глобальных растровых слоев, а `GET /api/geodata/project-rasters/by-layer-id/{id}` возвращает проектные растры.
* **Авторизация доступа к COG / Terrain:** 1. В момент инициализации растрового слоя в QGIS, плагин запрашивает у `geodata-service` доступ к оригиналу.
2. Микросервис проверяет права пользователя по JWT.
3. В случае успеха сервис генерирует временную подписанную ссылку (**Presigned URL**) к файлу в MinIO (с ограничением по времени жизни, например, 8 часов).
4. Ссылка передается в QGIS. Ядро GDAL осуществляет чтение COG по этой ссылке. MinIO самостоятельно валидирует подпись в HTTP Range-запросах, разгружая `auth-service`.

### 4.3. Аутентификация (Auth Service)

* Обмен учетных данных на JWT-токены. Плагин использует встроенный `QgsAuthManager` для безопасного хранения токенов в операционной системе картографа. JWT автоматически подставляется в заголовок `Authorization: Bearer <JWT>` каждого REST- и WebSocket-запроса.

## 5. Архитектура клиентского плагина (PyQGIS)

Плагин реализуется на языке Python с использованием библиотеки PyQt5/PyQt6 и внедряется в интерфейс QGIS в виде боковой панели (Dock Widget).

### Основные программные модули:

1. **Connection & Auth Manager:** Отвечает за авторизацию (OAuth2), обновление JWT-токенов по истечении срока их действия и поддержание стабильного WebSocket-соединения с Sync Service.
2. **Project Tree Explorer:** Визуализирует структуру папок и слоев GeoInfoSystem. Позволяет пользователю в один клик добавлять векторные (GeoJSON) или растровые (WMS/COG/Terrain) слои в текущую сессию QGIS.
3. **Feature Sync Engine:** Перехватывает события редактирования векторных слоев QGIS (`layer.editCommandCommitted`). Формирует JSON-пакеты изменений и отправляет их в фоновом потоке (`QgsTask`) в REST API, предотвращая фризы интерфейса.
4. **Advanced Raster Loader:** Управляет логикой загрузки растров. Переключает источники данных: при стандартном просмотре активирует WMS-версию слоя, при вызове инструментов анализа рельефа (уклоны, гидрология) автоматически подменяет источник на защищенный COG GeoTIFF через механизм Presigned URL.
5. **Direct Storage Uploader:** Позволяет загружать новые локальные растры и векторы в систему, запрашивая у бэкенда Presigned URL на `PUT`-запрос в MinIO и отправляя файл напрямую, минуя пропускные лимиты API Gateway.

## 6. Безопасность и разграничение прав (RBAC)

* **Уровень шлюза:** `api-gateway` производит базовую валидацию подписи JWT и его срока действия для каждого входящего запроса.
* **Уровень бизнес-логики:** Микросервисы `geodata` и `geo-abstraction` используют декларативную проверку прав (например, `@PreAuthorize("@rbac.canReadProject(#projectId)")`) для контроля доступа к конкретным объектам и папкам внутри проектов.
* **Уровень WebSocket:** При установке соединения клиент передает JWT в стартовом фрейме (CONNECT). Приложение разрывает соединение, если токен не прошел валидацию или истек.

## 7. Дорожная карта реализации (Roadmap)

### Этап 1: Инфраструктура и Безопасность (Сроки: Спринт 1-2)

* Разработка микросервиса **Sync Service** (интеграция Kafka $\rightarrow$ WebSockets).
* Настройка генерации временных подписанных ссылок (Presigned URLs) в MinIO для COG и Cesium Terrain.
* Создание скелета плагина QGIS с модулем авторизации через `QgsAuthManager`.

### Этап 2: Проводник и визуализация данных (Сроки: Спринт 3-4)

* Реализация панели **Project Tree Explorer** (отображение структуры папок проекта в QGIS).
* Реализация автоматического подключения WMS/WMTS слоев для базового просмотра.
* Настройка нативного отображения трехмерного рельефа через Cesium Terrain.

### Этап 3: Аналитические растры и Векторный CRUD (Сроки: Спринт 5-6)

* Интеграция подмодуля **Advanced Raster Loader** для работы с COG GeoTIFF без скачивания файлов. Тестирование растрового калькулятора QGIS на удаленных COG.
* Реализация двусторонней синхронизации векторов (точки, линии, полигоны) с поддержкой фильтрации по BBox.

### Этап 4: Реактивность и Оптимизация (Сроки: Спринт 7-8)

* Подключение плагина к Sync Service по WebSockets, обработка события `LAYER_UPDATED` для автообновления карты.
* Настройка агрессивного клиентского кэширования растровых тайлов для минимизации нагрузки на Nginx/MinIO.
* Покрытие модулей плагина интеграционными тестами.
