# Реализация перехода с GeoJSON на Vector Tiles (MVT) и интеграции `pg_tileserv`

## 1. Статус: Завершено ✅

Переход на MVT успешно реализован. Система использует `pg_tileserv` для высокопроизводительного рендеринга векторов. Основной режим отображения в веб-интерфейсе переведен на векторные тайлы, что позволило существенно повысить плавность работы карты при отображении крупных наборов данных (NDVI, сложные полигоны).

- резко снизить объем данных, передаваемых в браузер;
- убрать зависания и падения Chrome при отображении крупных векторных наборов;
- сохранить текущую микросервисную архитектуру и принцип `Database per Microservice`;
- оставить `Geo Data Service` как сервис управления данными, а не рендеринга;
- внедрить `pg_tileserv` как слой публикации тайлов из PostGIS с минимальными изменениями в текущей системе.

## 2. Почему текущая схема на GeoJSON не масштабируется

### Текущая модель
Сейчас `Geo Data Service` отдает клиенту векторы в формате GeoJSON. Это удобно для простых сценариев, но плохо масштабируется при больших слоях, особенно после импорта из QGIS векторизованных результатов анализа NDVI и аналогичных наборов.

### Основные проблемы
1. **Большой объем ответа**  
   GeoJSON передает все объекты слоя целиком, даже если на экране нужен только небольшой фрагмент.

2. **Высокая стоимость парсинга в браузере**  
   Chrome вынужден распарсить большой JSON, построить большое количество объектов и отрисовать их в одном кадре.

3. **Отсутствие зум-зависимой детализации**  
   Один и тот же детализированный набор объектов отправляется и на низких, и на высоких масштабах.

4. **Слабая переносимость к тяжелой геометрии**  
   После векторизации NDVI могут появляться тысячи мелких полигонов, сложные границы, внутренние кольца и большое число вершин.

5. **Рендеринг становится задачей клиентского устройства**  
   Вместо передачи компактных тайлов система перегружает браузер сырыми объектами.

### Вывод
GeoJSON остается приемлемым форматом для:
- экспорта;
- редактирования отдельных объектов;
- точечных выборок по одному объекту;
- отладочных сценариев.

Но для основного картографического отображения в веб-клиенте GeoJSON нужно заменить на MVT.

## 3. Целевая модель обмена данными

### Новая схема
**Клиент → Vector Tile Server → PostGIS**

Вместо выдачи GeoJSON сервисом `Geo Data Service`:

- `Geo Data Service` сохраняет CRUD, импорт, права, бизнес-логику и метаданные;
- `pg_tileserv` читает данные из PostGIS и отдает `.pbf` тайлы;
- клиент получает только ту геометрию, которая попадает в конкретный тайл и текущий zoom level.

### Что меняется концептуально
- вместо "вернуть все объекты слоя" используется "вернуть объекты только в рамках tile bounds";
- вместо одного большого ответа есть поток маленьких тайлов;
- вместо одинаковой детализации для всех масштабов — зум-зависимая детализация;
- вместо тяжёлого GeoJSON — компактный бинарный формат MVT.

## 4. Рекомендуемая целевая архитектура

### Роль компонентов

#### `Geo Data Service`
Остается:
- владельцем бизнес-логики данных;
- точкой импорта из QGIS и других источников;
- API для CRUD-операций;
- источником событий для Kafka;
- местом, где сохраняются метаданные, доступы, папки, статусы объектов.

Не должен:
- формировать большие GeoJSON для основного рендеринга карты;
- быть рендеринговым API для тяжелых слоев.

#### `pg_tileserv`
Становится:
- отдельным read-only tile service;
- читает только опубликованные spatial tables / views;
- отдает MVT для визуализации в веб-клиенте.

#### Frontend
Переходит на:
- `vector tile source`;
- стилизацию слоев на стороне клиента;
- запрос тайлов по зуму и видимой области;
- получение атрибутов через `feature id` и отдельные запросы при необходимости.

#### `nginx-proxy`
Становится:
- внешней точкой входа для `pg_tileserv`;
- местом для CORS, кэширования и контроля заголовков;
- промежуточным кэшем для `.pbf`-ответов.

## 5. Почему именно `pg_tileserv` в текущем этапе

Для вашей текущей системы `pg_tileserv` подходит как первый этап миграции, потому что:

- его проще встроить в существующий Docker Compose;
- он напрямую работает с PostGIS и не требует отдельной схемы публикации;
- он хорошо подходит для быстрого перевода текущих таблиц и VIEW в MVT;
- он позволяет оставить `Geo Data Service` без радикальной переработки;
- он закрывает именно проблему "слишком тяжёлый GeoJSON в браузере".

### Когда `pg_tileserv` особенно хорош
- нужен быстрый переход без смены всего геостека;
- слой уже хранится в PostGIS;
- нужен простой и прозрачный маршрут `PostGIS → Tile Server → Browser`;
- важна минимальная стоимость внедрения.

### Ограничение подхода
Если в будущем потребуется:
- публикация из нескольких источников;
- более сложная серверная упаковка тайлов;
- дополнительные механизмы генерации/кеширования;
- статические MVT/PMTiles/MBTiles пайплайны,

тогда можно рассматривать эволюцию в сторону более богатого tile-сервера, но **не в первый этап миграции**.

## 6. Изменения в БД

Вы уже изменили модель:
- `Point` → `MultiPoint`
- `Polygon` → `MultiPolygon`

Это правильное решение для унификации геометрий.

### Почему это полезно
- снимает проблемы смешанных геометрий;
- упрощает публикацию в MVT;
- делает слой предсказуемым для tile pipeline;
- уменьшает число исключений при импорте из QGIS;
- улучшает дальнейшую обработку `ST_Multi`, `ST_CollectionExtract`, `ST_Dump`.

### Обязательные требования к геометриям
- единый SRID;
- единый тип геометрии на таблицу;
- валидная геометрия;
- наличие spatial index;
- желательно хранить `id` как неизменяемый ключ объекта.

### Рекомендуемые индексы
Для всех spatial таблиц:

```sql
CREATE INDEX IF NOT EXISTS ix_project_points_geom
  ON geodata.project_points USING GIST (geom);

CREATE INDEX IF NOT EXISTS ix_project_multilines_geom
  ON geodata.project_multilines USING GIST (geom);

CREATE INDEX IF NOT EXISTS ix_project_polygons_geom
  ON geodata.project_polygons USING GIST (geom);
```

Дополнительно, если часто используются фильтры:
- `project_id`
- `folder_id`
- `status`
- `created_date`

то стоит добавить btree-индексы на эти поля.

## 7. Главный принцип миграции: разделить данные на два контура

### Контур 1. Операционный
Используется `Geo Data Service`:
- редактирование;
- импорт;
- удаление;
- бизнес-проверки;
- события;
- связь с папками и проектами.

### Контур 2. Публикационный
Используется `pg_tileserv`:
- только чтение;
- публикация только нужных слоев;
- упрощение геометрий;
- адаптация к zoom level;
- контроль атрибутов, попадающих в тайл.

Это ключевой момент: **не нужно напрямую публиковать "сырые" таблицы как есть**. Для MVT лучше использовать отдельные VIEW или materialized VIEW.

## 8. Рекомендуемая стратегия публикации данных для MVT

### Базовый подход
Создать отдельные представления для публикации:

- `*_mvt` или `*_tile`
- либо `v_*_mvt`

Например:
- `geodata.project_points_mvt`
- `geodata.project_multilines_mvt`
- `geodata.project_polygons_mvt`

### Что должно быть в MVT-view
Только:
- `id`
- `geom`
- нужные свойства для стиля и фильтрации

Не нужно тащить в тайл:
- большие текстовые поля;
- служебные поля;
- всё, что не отображается на карте.

### Базовая идея SQL
```sql
CREATE OR REPLACE VIEW geodata.project_polygons_mvt AS
SELECT
    id,
    project_id,
    folder_id,
    name,
    status,
    ST_Multi(geom) AS geom
FROM geodata.project_polygons
WHERE geom IS NOT NULL;
```

### Для тяжелых слоев
Если слой очень плотный, лучше делать не просто `VIEW`, а:

- `VIEW` с упрощением;
- `MATERIALIZED VIEW` для низких zoom;
- отдельные представления по zoom-группам.

## 9. Стратегия для NDVI и больших полигональных наборов

NDVI и другие векторизованные растровые результаты часто дают:
- много мелких полигонов;
- сложные границы;
- тысячи фич в пределах одного экрана;
- резкий рост веса слоя на низких зумах.

### Для таких слоев рекомендуется

#### 9.1. Разделение на уровни детализации
Сделать несколько представлений:
- `ndvi_z0_7`
- `ndvi_z8_11`
- `ndvi_z12_plus`

На низких зумах показывать:
- агрегированные или упрощенные полигоны;
- классифицированные зоны;
- не все исходные пиксельные границы.

#### 9.2. Упрощение геометрии
Использовать:
- `ST_SimplifyPreserveTopology`
- `ST_SnapToGrid`
- `ST_Subdivide`
- `ST_MakeValid` при необходимости

#### 9.3. Декомпозиция слишком больших геометрий
Если один объект очень большой:
- разбить его на более мелкие фрагменты;
- либо сформировать отдельный материализованный слой для визуализации.

#### 9.4. Не отдавать весь NDVI как есть
Для карты важнее:
- цветовая зона;
- класс NDVI;
- общая форма;
- интерактивный hover/click.

Полная геометрическая точность нужна только в отдельном просмотре объекта.

## 10. Рекомендованная схема перехода от GeoJSON к MVT

### Этап 1. Подготовка БД
- привести все `Point` к `MultiPoint`;
- привести все `Polygon` к `MultiPolygon`;
- проверить SRID;
- создать / проверить GiST индексы;
- подготовить публикационные VIEW.

### Этап 2. Поднять `pg_tileserv`
- развернуть контейнер;
- подключить к той же PostGIS;
- ограничить доступ только на чтение;
- публиковать только нужную схему `geodata` или даже только VIEW.

### Этап 3. Включить двойной режим
Параллельно держать:
- старый GeoJSON endpoint;
- новый MVT endpoint.

На этом этапе фронтенд можно переключать флагом:
- старый путь для fallback;
- новый путь для тестирования и части пользователей.

### Этап 4. Перевести Frontend на tiles
Вынести:
- базовые слои;
- тематические слои;
- NDVI-слои;
- выборку объектов.

### Этап 5. Отключить GeoJSON для тяжелых слоев
GeoJSON оставить только для:
- точечного экспорта;
- отладки;
- очень малых выборок;
- отдельных API вызовов, где это оправдано.

- использование GeoJSON как основного картографического протокола.

- endpoint для регистрации `mvt layer source`;
- endpoint для получения конфигурации стиля слоя;
- при необходимости — endpoint для feature info по id.

## 13. Что нужно изменить во frontend

### Было
- запрос GeoJSON;
- ручной рендер большого FeatureCollection;
- большое количество объектов в памяти;
- риск падения Chrome.

### Станет
- источник vector tiles;
- загрузка только видимых тайлов;
- рендер только того, что видно на экране;
- меньше памяти и меньше нагрузка на GC.

### Практические изменения
1. **Заменить GeoJSON source на MVT source**
2. **Настроить стили по zoom**
3. **Оставить отдельный запрос по id для pop-up / details**
4. **Сделать hover и selection через feature id**
5. **Для NDVI использовать тематический слой с малым числом свойств**

### Важное замечание по OpenLayers / Cesium
- Для 2D-карты MVT подходит напрямую.
- Для Cesium MVT не должен быть главным форматом для тяжелого массового рендеринга; для 3D лучше сохранять отдельный 3D-стек или показывать только ограниченные слои.

## 14. Как должен выглядеть `pg_tileserv` в Docker Compose

```yaml
services:
  postgis:
    image: postgis/postgis:16-3.4
    container_name: postgis
    environment:
      POSTGRES_DB: geodata_db
      POSTGRES_USER: tileserver
      POSTGRES_PASSWORD: tileserver_pass
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data

  pg_tileserv:
    image: crunchydata/pg_tileserv:latest
    container_name: pg_tileserv
    environment:
      TS_DATABASE_URL: postgres://tileserver:tileserver_pass@postgis:5432/geodata_db
      TS_HTTP_PORT: 7800
      TS_CORS_ORIGINS: "*"
      TS_MAX_CONNECTIONS: 10
    ports:
      - "7800:7800"
    depends_on:
      - postgis

  nginx:
    image: nginx:stable
    container_name: nginx
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - pg_tileserv

volumes:
  pgdata:
```

## 15. Пример `nginx`-прокси для tile server

```nginx
server {
    listen 80;

    location /tiles/ {
        proxy_pass http://pg_tileserv:7800/;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        add_header Cache-Control "public, max-age=60";
        add_header Access-Control-Allow-Origin "*";
        add_header Access-Control-Allow-Headers "*";
    }
}
```

### Зачем это нужно
- единая точка доступа;
- можно включить кэширование;
- можно повесить авторизацию;
- можно скрыть внутренние адреса контейнеров;
- проще контролировать лимиты и заголовки.

## 16. Безопасность

### Минимальный безопасный вариант
Создать отдельного пользователя БД, например `tileserver`, с правами только на `SELECT`:

- только на нужную схему;
- только на нужные VIEW / таблицы;
- без `INSERT`, `UPDATE`, `DELETE`.

### Более строгий вариант
Разрешить `pg_tileserv` видеть только:
- публичные слои;
- агрегированные слои;
- слои, не завязанные на права конкретного пользователя.

### Если нужны приватные слои
Есть три пути:

1. **Проксирование через ваш gateway**
   - `pg_tileserv` не открыт напрямую;
   - доступ контролирует `nginx`/gateway.

2. **RLS на стороне Postgres**
   - сложнее, но безопаснее;
   - хорошо подходит для фильтрации по `project_id`/`user`.

3. **Отдельные tile-слои по проектам**
   - проще в эксплуатации;
   - удобнее для разграничения прав.

Для первого этапа рекомендую:
- публичные / общие слои через `pg_tileserv`;
- приватные объекты оставить в API `Geo Data Service`.

## 17. Кэширование и производительность

### Что кэшировать
- TileJSON;
- `.pbf`-тайлы;
- ответы на часто используемые zoom/extent комбинации.

### Где кэшировать
- nginx reverse proxy;
- при необходимости внешний HTTP cache;
- при росте нагрузки — отдельный cache layer.

### Что дополнительно помогает
- ограничение `max features per tile`;
- генерализованные представления;
- `ST_Subdivide` для больших полигонов;
- `ST_SimplifyPreserveTopology`;
- ограничение свойств, попадающих в тайл;
- GiST по `geom`;
- btree по фильтрам.

## 18. Набор SQL-представлений для публикации

### Точки
```sql
CREATE OR REPLACE VIEW geodata.v_project_points_mvt AS
SELECT
    id,
    project_id,
    folder_id,
    name,
    status,
    geom
FROM geodata.project_points
WHERE geom IS NOT NULL;
```

### Линии
```sql
CREATE OR REPLACE VIEW geodata.v_project_multilines_mvt AS
SELECT
    id,
    project_id,
    folder_id,
    name,
    status,
    geom
FROM geodata.project_multilines
WHERE geom IS NOT NULL;
```

### Полигоны
```sql
CREATE OR REPLACE VIEW geodata.v_project_polygons_mvt AS
SELECT
    id,
    project_id,
    folder_id,
    name,
    status,
    ST_Multi(geom) AS geom
FROM geodata.project_polygons
WHERE geom IS NOT NULL;
```

### NDVI-слой с упрощением
```sql
CREATE OR REPLACE VIEW geodata.v_ndvi_mvt AS
SELECT
    id,
    project_id,
    ndvi_class,
    ST_SimplifyPreserveTopology(geom, 0.0001) AS geom
FROM geodata.ndvi_polygons
WHERE geom IS NOT NULL;
```

### Когда нужны материализованные представления
Если слой:
- очень большой;
- почти не меняется;
- дорогой для постоянного вычисления;

тогда лучше `MATERIALIZED VIEW` и регламентное обновление.

## 19. Стратегия публикации слоев в `pg_tileserv`

### Рекомендуется публиковать:
- только нужные VIEW;
- только нужную схему;
- только `SELECT`-доступные объекты.

### Не рекомендуется публиковать:
- все системные таблицы;
- сырые рабочие таблицы без фильтрации;
- таблицы с тяжелыми техническими полями;
- объекты, не предназначенные для визуализации.

### Хорошая практика
Сделать отдельную схему, например:
- `geodata_publish`

И публиковать только ее.

- выделение самых тяжелых слоев;
- фиксация проблемных сценариев Chrome.

### Этап 1. Подготовка БД
- окончательно унифицировать геометрии;
- проверить валидность;
- поставить индексы;
- создать публикационные VIEW.

### Этап 2. Развернуть `pg_tileserv`
- поднять контейнер;
- подключить к PostGIS;
- ограничить права;
- проверить каталог слоев;
- проверить отдачу тайлов.

### Этап 3. Подключить frontend в тестовом режиме
- новый MVT-слой включить по feature flag;
- проверить панораму карты;
- проверить выборку объектов;
- проверить popup и highlight.

### Этап 4. Перевести самые тяжелые слои
- NDVI;
- плотные полигоны;
- слои с тысячами объектов на экран.

### Этап 5. Перевести остальные векторные слои
- точки;
- линии;
- вспомогательные тематические слои.

### Этап 6. Ограничить GeoJSON
- оставить только для редких выборок;
- убрать из основного картографического пути.

## 21. План отката

Откат должен быть простым:

1. выключить MVT-флаг на frontend;
2. вернуть GeoJSON endpoint как основной;
3. оставить `pg_tileserv` в standby;
4. при необходимости откатить SQL VIEW и конфиги;
5. восстановить БД из backup, если изменения затронули данные.

Главное правило: **новый tile pipeline не должен ломать старый путь до полной верификации**.

---

## 22. Критерии успешности перехода

Переход можно считать успешным, если:

- браузер перестал зависать на тяжелых слоях;
- средний размер ответа заметно снизился;
- скорость первого отображения карты выросла;
- NDVI и крупные полигоны рисуются стабильно;
- `Geo Data Service` продолжает выполнять CRUD;
- `pg_tileserv` отдает тайлы без ручной постобработки на клиенте;
- инфраструктура осталась простой и поддерживаемой.

---

## 23. Рекомендуемое решение

### Для текущего этапа:
**Внедрить `pg_tileserv` как отдельный read-only MVT слой поверх PostGIS и перевести основные картографические слои с GeoJSON на vector tiles.**

### Почему это лучший первый шаг
- минимально вмешивается в текущую микросервисную архитектуру;
- решает основную проблему перегрузки браузера;
- позволяет сохранить `Geo Data Service` без переделки бизнес-логики;
- дает быстрый выигрыш по производительности;
- оставляет место для будущего расширения tile-архитектуры.

---

## 24. Итог

Переход на MVT нужно рассматривать не как смену формата, а как смену принципа доставки геоданных:

- **GeoJSON** — для точечных операций и экспорта;
- **MVT** — для основного картографического отображения;
- **Geo Data Service** — для управления данными;
- **pg_tileserv** — для публикации тайлов;
- **PostGIS** — как единый источник истины;
- **nginx** — как кэш и точка внешнего доступа.

Это даст более устойчивую, быструю и масштабируемую карту без падений браузера на больших слоях.
