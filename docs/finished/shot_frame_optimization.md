# **Реализованное решение: Оптимизация редактирования тяжелых векторов (Shot Frame / MVT) V2**

Настоящий документ описывает финальную архитектуру и реализацию стратегии оптимизации работы с тяжелыми пространственными данными, внедренную в систему.

## 1. Проблема
При редактировании объектов с экстремально сложной геометрией (десятки тысяч вершин, сотни островов в мультиполигонах) классический подход с полной загрузкой GeoJSON приводил к:
- Зависанию UI-потока браузера.
- Избыточному потреблению оперативной памяти.
- Огромному сетевому трафику при сохранении одной вершины.

## 2. Архитектура "Shot Frame" (Кадр)

Вместо загрузки всего объекта, система теперь работает только с видимой областью карты.

### 2.1. Ограничение по Zoom
Редактирование мульти-геометрий разрешено только при достижении `VITE_SHOT_FRAME_MIN_ZOOM` (по умолчанию 16). Это гарантирует, что пользователь видит детализированную часть объекта, и "кадр" содержит разумное количество данных.

### 2.2. Динамическая подгрузка (Backend)
В `geodata-service` реализованы эндпоинты для извлечения фрагментов:
- **SQL-логика**: Используется `ST_Dump` для декомпозиции мульти-геометрии и `ST_Intersects` для фильтрации по BBox.
- **Совместимость**: Реализация выполнена на базовых функциях PostGIS без использования `ST_SetGeometryN`, что обеспечивает поддержку старых версий расширения.
- **Размерность**: При сохранении используется `ST_Force3D`, что гарантирует целостность Z-координаты.

### 2.3. Интерактив (Frontend)
- **Визуализация**: Добавлена визуальная рамка (Shot Frame) с отступом 15px от краев карты.
- **Debounce**: Запросы к API выполняются с задержкой 1с после остановки карты (`moveend`), чтобы исключить лишнюю нагрузку при скроллинге.
- **Атомарные правки**: Фронтенд отслеживает только измененные `sub_id` фрагментов и отправляет на сервер только их через метод `PATCH`.

## 3. Решение проблем кэширования

### 3.1. Browser Cache Buster
К URL векторных тайлов (MVT) добавлен динамический параметр `t=${timestamp}`. Он обновляется автоматически при каждом успешном сохранении правок, что заставляет браузер запросить свежие тайлы.

### 3.2. Nginx Proxy Invalidation
Конфигурация Nginx обновлена для включения параметров строки запроса в ключ кэширования:
```nginx
proxy_cache_key "$host$request_uri";
```
Это гарантирует, что после изменения `cacheBuster` на фронтенде, Nginx также воспримет запрос как новый (Cache MISS) и проксирует его к `pg_tileserv`.

## 4. Инструкция по настройке
В файле `.env` доступны следующие параметры:
- `VITE_SHOT_FRAME_MIN_ZOOM`: Минимальный уровень приближения для активации режима редактирования.

## 5. Итоги
Внедрение стратегии позволило:
1. Редактировать объекты любого размера (хоть границы целой страны).
2. Снизить нагрузку на сеть на 90%+ при работе с тяжелыми векторами.
3. Обеспечить мгновенную синхронизацию карты у всех пользователей за счет корректной инвалидации кэша.
