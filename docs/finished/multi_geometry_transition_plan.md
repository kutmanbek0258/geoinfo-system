# Стратегия перехода к Multi-Geometry (GeoInfoSystem)

## 1. Цель
Обеспечить полную совместимость системы с профессиональными ГИС (QGIS, ArcGIS) путем перехода от одиночных типов геометрии (`Point`, `Polygon`) к составным (`MultiPoint`, `MultiPolygon`). Это позволит импортировать сложные объекты (группы точек, полигоны с отверстиями, мульти-полигоны) без потери данных и ошибок типизации.

## 2. Этапы реализации

### Этап 1: База данных (PostGIS)
Необходимо изменить типы геометрии в существующих таблицах. Использование функции `ST_Multi` гарантирует сохранение текущих данных.

- **Таблица `project_points`**:
  - Тип: `geometry(PointZ, 4326)` -> `geometry(MultiPointZ, 4326)`
- **Таблица `project_polygons`**:
  - Тип: `geometry(PolygonZ, 4326)` -> `geometry(MultiPolygonZ, 4326)`
- **Таблица `project_multilines`**: Оставить без изменений (`MultiLineStringZ`).

### Этап 2: Backend (Java / geodata-service)
Обновление серверной логики для работы с составными объектами.

- **Модели (Entities)**:
  - `ProjectPoint`: замена `org.locationtech.jts.geom.Point` на `MultiPoint`.
  - `ProjectPolygon`: замена `org.locationtech.jts.geom.Polygon` на `MultiPolygon`.
- **DTO**:
  - Обновить типы во всех DTO (`Create...`, `Update...`, `Project...Dto`).
- **Мапперы**:
  - Проверка корректности маппинга через MapStruct.

### Этап 3: Плагин QGIS (Python)
Автоматизация подготовки данных перед отправкой в API.

- **Нормализация**:
  - Внедрить метод `convertToMultiType()` для всех слоев.
  - Удалить устаревшую логику извлечения "первой части" (`next(geom.parts())`), которая приводила к потере данных.
- **Стабильность**: Обеспечить корректную передачу Z-координаты.

### Этап 4: Верификация
- Тестовый импорт из QGIS объектов с дырками (Donut polygons).
- Тестовый импорт групп точек (MultiPoint).
- Проверка отображения на фронтенде (OpenLayers/Cesium).
