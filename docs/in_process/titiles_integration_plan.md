### **Стратегия миграции подсистемы растровых данных: GeoServer ➔ TiTiler**

#### **1. Анализ текущего состояния (As-Is) и мотивация**

* **Текущая маршрутизация:** GeoServer используется исключительно для растров (GeoTIFF) и отдает их по WMS/WMTS.
* **Хранение:** Данные лежат в S3-совместимом хранилище MinIO.
* **Проблема:** GeoServer не предназначен для прямой (stateless) работы с S3. Требуется синхронизация состояния, публикация слоев через REST API и нагрузка на CPU при рендеринге.
* **Цель:** Заменить stateful-сервер (GeoServer) на stateless-тайлер (TiTiler), который читает Cloud Optimized GeoTIFF (COG) напрямую из MinIO "на лету".

---

#### **2. Пошаговый план перехода**

**Этап 1: Адаптация конвейера обработки данных (GeoAbstract Worker)**
В текущей архитектуре `GeoAbstract Worker` уже отвечает за обработку снимков (например, Sentinel-2) и создание COG.

* **Действие:** Убедиться, что скрипты Python/GDAL в воркере генерируют строго валидные COG-файлы с внутренними пирамидами (overviews) и тайлингом.
* **Действие:** Прекратить вызовы REST API GeoServer для публикации слоя после сохранения файла в MinIO. Воркер должен только публиковать событие завершения в топик `geoabstraction.raster.events`.

**Этап 2: Изменения в БД и Geo Data / Geo Abstraction Service**
Вместо хранения названия опубликованного слоя (Workspace:LayerName), системе нужно хранить только S3-ключ.

* **Действие:** В таблице `Imagery Layers` изменить структуру метаданных. Вместо URL WMS сервера хранить относительный путь в MinIO (например, `s3://geodata/sentinel2/scene_1.tif`).
* **Действие:** Вынести конфигурацию стилей (цветовые шкалы, рескейлинг) в JSON-поле в БД, чтобы API передавал эти параметры на фронтенд.

**Этап 3: Развертывание и настройка Инфраструктуры (Docker Compose)**

* **Действие:** Удалить контейнер GeoServer из `docker-compose.yml`.
* **Действие:** Добавить сервис `titiler` на базе `ghcr.io/developmentseed/titiler`.
* **Действие:** Настроить переменные окружения TiTiler (`AWS_ACCESS_KEY_ID`, `AWS_S3_ENDPOINT` и т.д.) для связи с локальным контейнером MinIO.
* **Действие:** Обновить `nginx-proxy` (который выступает внешним шлюзом), добавив location для маршрутизации `/api/raster/` на порт TiTiler. Настроить на уровне nginx кэширование XYZ-тайлов, аналогично тому, как сейчас кэшируются MVT.

**Этап 4: Адаптация Фронтенда (Vue.js + OpenLayers / Cesium)**

* **Действие:** Перевести растровые слои в OpenLayers с использования классов `ImageWMS` или `TileWMS` на `XYZ`.
* **Действие:** Реализовать функцию-билдер, которая конструирует URL к шлюзу `nginx-proxy` `/api/raster/cog/tiles/...`, подставляя S3 URI из метаданных слоя и параметры стилизации.
* **Действие:** В Cesium переключиться на `UrlTemplateImageryProvider`.

**Этап 5: Адаптация модуля печати (Geo Print Service)**
Это наиболее критичный узел при отказе от WMS. В текущей схеме `Geo Print Service` использует GeoTools для рендеринга WMS-подложек и векторов.

* **Действие:** Поскольку GeoTools отлично работает с WMS, но не имеет "нативной" поддержки динамических XYZ-тайлов из S3, необходимо изменить логику работы `MapRenderer`.
* **Решение:** TiTiler имеет эндпоинт `/preview` или `/bbox`, который позволяет запросить единое изображение (PNG/JPEG) по заданным координатам BBox и разрешению экрана. `Geo Print Service` должен запрашивать у TiTiler это цельное изображение для нужного охвата печати и использовать его как фоновый слой `BufferedImage` перед наложением GeoJSON-объектов.

#### **3. Архитектурные изменения (До и После)**

| Компонент | Как сейчас (С GeoServer) | Как будет (С TiTiler) |
| --- | --- | --- |
| **Публикация данных** | Worker ➔ Сохранение в MinIO ➔ REST запрос к GeoServer для создания слоя. | Worker ➔ Сохранение в MinIO. (Всё, данные уже доступны). |
| **Хранение стейта** | В PostGIS + внутренний каталог (Data Directory) GeoServer. | Только в PostGIS (MinIO URL и настройки стиля). TiTiler — полностью Stateless. |
| **Запрос карты (Фронтенд)** | Браузер ➔ `nginx-proxy` ➔ GeoServer (рендерит на лету из S3) ➔ Браузер. | Браузер ➔ `nginx-proxy` ➔ TiTiler (стримит куски напрямую из S3) ➔ Браузер. |
| **Стилизация** | SLD (XML), хранится и обрабатывается на GeoServer. | Query Parameters (URL), формируются на фронтенде Vue.js динамически. |
| **Печать PDF (GeoTools)** | Прямой GetMap запрос к WMS GeoServer'а. | Запрос статической картинки по BBox через API TiTiler. |

#### **4. Итоги и Риски миграции**

**Преимущества:**

1. **Снижение потребления RAM:** TiTiler потребляет в разы меньше памяти, чем Java-основанный GeoServer.
2. **Идеальное разделение ответственности (SoC):** Теперь микросервисная архитектура полностью очищена от stateful-геосерверов. Все настройки стилей живут в БД (или фронтенде), а не в разрозненных XML-файлах внутри файловой системы стороннего сервиса.
3. **Ускорение развертывания:** Отсутствие необходимости синхронизировать каталог данных GeoServer делает CI/CD и горизонтальное масштабирование бэкенда тривиальной задачей.

**Основные риски:**

* **Рефакторинг Geo Print Service:** Потребуется переписать логику формирования подложки в классе-рендерере отчетов, так как уходит стандартный OGC WMS клиент.
* **Прогрев кэша:** В отличие от GeoWebCache, кэширование ляжет исключительно на плечи `nginx-proxy`. Потребуется тонкая настройка `proxy_cache` для маршрутов TiTiler.
