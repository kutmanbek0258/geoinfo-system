# 🎨 Архитектура стилизации растров (ранее SLD)

Система перешла от XML-файлов стилей SLD (Styled Layer Descriptor), обрабатывавшихся на стороне GeoServer, к гибкой модели **базы данных и динамического рендеринга на стороне TiTiler**.

Все старые файлы `.sld` удалены из ресурсов бэкенда. Вместо них стили хранятся в таблице `raster_styles` в формате JSONB и редактируются через визуальный редактор на фронтенде.

---

## 💾 Хранение стилей в БД (JSONB)

Таблица `raster_styles` находится в схеме `geodata` базы данных `geodata_db` и имеет следующую структуру:

| Колонка | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ (генерируется автоматически). |
| `name` | VARCHAR(100) | Уникальное кодовое имя стиля (например, `vegetation_index`, `ndwi_water`). |
| `title` | VARCHAR(255) | Понятное пользователю название стиля. |
| `type` | VARCHAR(50) | Тип отрисовки (`ramp` - градиент, `values` - дискретный, `discrete`). |
| `config` | JSONB | Массив объектов точек интерполяции. |
| `is_system` | BOOLEAN | Флаг системного стиля (`true` запрещает редактирование и удаление стиля). |
| `created_date` | TIMESTAMP | Дата создания стиля. |
| `last_modified_date` | TIMESTAMP | Дата последнего изменения стиля. |

### Структура JSONB в `config`
Массив объектов, каждый из которых содержит:
* `color`: HEX-код цвета (например, `#0000FF`).
* `quantity`: Числовое значение пикселя растра, к которому привязан цвет.
* `opacity`: Прозрачность цвета от `0.0` (прозрачно) до `1.0` (непрозрачно).
* `label`: Описание шага для легенды (например, "Вода / Снег").

Пример JSON-конфигурации:
```json
[
  { "color": "#000000", "quantity": -9999.0, "opacity": 0.0, "label": "NoData" },
  { "color": "#0000FF", "quantity": -1.0, "opacity": 1.0, "label": "Вода / Снег" },
  { "color": "#E29B63", "quantity": 0.0, "opacity": 1.0, "label": "Суша" },
  { "color": "#328846", "quantity": 1.0, "opacity": 1.0, "label": "Здоровая растительность" }
]
```

## 🚀 Динамический рендеринг на TiTiler

Вместо WMS-запросов к GeoServer, фронтенд строит URL-шаблоны XYZ тайлов и передает цветовую схему в параметре `colormap` в виде JSON-encoded интервалов:

`[[[min, max], [R, G, B, A]], ...]`

Фронтенд автоматически конвертирует JSONB-конфигурацию стиля в интервалы с помощью функции `buildTiTilerColormap` в утилите [titiler-style-builder.ts](file:///home/kutman/Documents/dev/geoinfo-system/frontend/src/util/titiler-style-builder.ts).

## 🎨 Предустановленные системные стили

В систему автоматически загружаются (seed через Liquibase) следующие 8 базовых стилей, заменяющие старые файлы SLD:

1. **`raster`** (Default Raster Style) — отображение исходного растра без наложения цветовых шкал.
2. **`vegetation_index`** — единая шкала для индексов биомассы (NDVI/SAVI/EVI). Переход от песочного цвета (суша, застройка) к темно-зеленому (густой лес).
3. **`ndwi_water`** — шкала для индекса открытой воды NDWI. Выделяет переувлажненные зоны и глубокие водоемы от голубого до темно-синего.
4. **`gndvi`** — шкала для агроаналитики и активности хлорофилла.
5. **`nbr_burn`** — шкала оценки гарей (Normalized Burn Ratio) для мониторинга лесных пожаров и восстановления растительности.
6. **`ndbi_urban`** — шкала индекса городской застройки (NDBI). Наглядно отделяет застройку (красный/бордовый) от растительности и воды.
7. **`ndmi_moisture`** — шкала индекса влажности растений (NDMI).
8. **`ndsi_snow`** — шкала индекса снега и льда (NDSI).

## 💻 Редактор стилизации во фронтенде

Для пользователей реализован графический интерфейс (Диалог `RasterStyleEditorDialog.vue`), позволяющий:
1. Выбирать существующие стили.
2. Создавать новые кастомные стили (клонированием системных или с нуля).
3. Интерактивно изменять шаги градиента с помощью **Vuetify Color Picker** (`v-color-picker`), редактировать числовые границы (`quantity`), прозрачность (`opacity`) и подписи (`label`).
4. Изменения сохраняются в базу данных `geodata_db` через REST API (`/api/geodata/raster-style`).
